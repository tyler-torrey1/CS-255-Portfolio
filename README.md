# DriverPass System Design Project
## Overview

This repository contains two artifacts from the DriverPass system analysis and design project completed as part of my coursework in software development lifecycles and system design. The goal of this project was to analyze a client's needs and design a software system that addresses those requirements through structured documentation and UML modeling.

The client, DriverPass, is a company that wanted to help students better prepare for their driving license exams. Their goal was to create a system that would allow customers to access online practice exams, schedule in-person driving lessons, and track their progress through training materials. The system also needed administrative functionality so employees could manage lesson schedules, maintain training packages, and monitor student performance.

Two key documents are included in this repository:

Business Requirements Document (BRD) – outlines the client’s needs, system goals, stakeholders, and functional requirements.

System Design Document – includes UML diagrams and technical specifications that describe how the DriverPass system would be implemented.

Together, these documents demonstrate the process of moving from requirements gathering to system design.

## What I Did Well

One of the strongest parts of my work in this project was translating business needs into structured system documentation. I focused on organizing the requirements clearly and ensuring that each component of the system design could be traced back to a specific user need. Creating UML diagrams such as activity diagrams, sequence diagrams, and class diagrams helped illustrate how different parts of the system interact and how user actions flow through the system.

I also worked to keep the documentation readable for both technical and nontechnical audiences, which is an important skill when communicating with clients or stakeholders who may not have a technical background.

## What I Would Improve

If I were to revise one part of the project, I would spend more time refining the UML class diagram and expanding the relationships between system entities. While the diagram captures the major components of the system, further iteration could improve how responsibilities and dependencies between classes are represented.

In a real development environment, this diagram would likely go through multiple design reviews to ensure the architecture is scalable and clearly communicates the system structure to developers.

## Interpreting User Needs

Understanding the user's needs was a critical part of this project. I analyzed the client interview and business requirements to determine what features were necessary for both customers and employees. From there, I translated those needs into system functionality such as user authentication, lesson scheduling, progress tracking, and administrative management tools.

Considering the user’s needs is essential when designing software because the success of a system depends on how well it supports the people who use it. If designers focus only on technical implementation without understanding the user experience, the resulting system may technically work but fail to solve the real problem.

## My Approach to Software Design

When approaching system design, I start by identifying the key stakeholders and understanding the goals of the system. From there, I break the system down into smaller components and model how those components interact.

Some techniques I use during the design process include:

Requirements analysis to understand what the system must accomplish

Use case modeling to identify how users interact with the system

UML diagrams such as activity, sequence, and class diagrams to visualize system behavior and structure

Iterative refinement, improving designs as new information or feedback becomes available

In future projects, I would continue to rely on these strategies while also incorporating additional design reviews and feedback loops to further refine system architecture.
